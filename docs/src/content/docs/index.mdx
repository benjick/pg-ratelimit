## Why pg-ratelimit?

- **You already have Postgres** - no need to add Redis or another service just for rate limiting
- **One fewer dependency** - zero runtime deps beyond `pg`, which you're already using
- **Durable mode** - survive crashes with logged tables for billing/quota use cases
- **Postgres handles it fine** - for most apps, Postgres can easily handle the rate-limiting write load alongside your application queries
- **UNLOGGED closes the perf gap** - ephemeral (UNLOGGED) tables skip WAL writes, making throughput competitive with in-memory stores
